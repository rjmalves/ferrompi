# ADR-0001: Hand-written C Wrapper Layer

**Status:** Accepted — 2026-05-17
**Date:** 2026-05-17
**Deciders:** Rogerio Alves

---

## Context

The most common question from new contributors is: _why does ferrompi
maintain a hand-written C layer at `csrc/ferrompi.c` (4629 LOC) and
`csrc/ferrompi.h` (2171 LOC) instead of generating Rust bindings directly
from MPI headers via `bindgen`?_ Most existing Rust MPI bindings — notably
`rsmpi` — take exactly that approach: run `bindgen` over the system MPI
headers, produce a `*-sys` crate with raw Rust types, and layer safe
wrappers on top. ferrompi did not. This ADR is the authoritative answer.

ADR-0001 is numbered to reflect logical precedence. The handle-table
strategy (ADR-0002, `docs/adr/0002-handle-tables.md`) and the op-create
trampoline design (ADR-0005, `docs/adr/0005-mpi-op-create.md`) both presuppose
the existence and structure of this C wrapper. ADR-0001 documents the
foundation that those later records build on.

### What the C layer does today

`csrc/ferrompi.c` is responsible for six categories of work:

1. **Handle tables.** The C layer maintains seven fixed-size static tables
   mapping opaque MPI handle types to integer slot indices that Rust can
   store and compare safely:

   | Table            | Capacity | MPI handle type |
   | ---------------- | -------- | --------------- |
   | `comm_table`     | 256      | `MPI_Comm`      |
   | `request_table`  | 16384    | `MPI_Request`   |
   | `win_table`      | 256      | `MPI_Win`       |
   | `info_table`     | 64       | `MPI_Info`      |
   | `group_table`    | 64       | `MPI_Group`     |
   | `datatype_table` | 64       | `MPI_Datatype`  |
   | `op_table`       | 16       | `MPI_Op`        |

   Each table has a parallel occupancy array and an `alloc_*`/`free_*`/`get_*`
   function triple. The request table uses C11 atomics for thread safety under
   `MPI_THREAD_MULTIPLE`; see ADR-0002 (`docs/adr/0002-handle-tables.md`) for
   the full CAS strategy. Rust callers never hold a raw MPI handle; they hold
   an `int32_t` slot index.

2. **Large-count `_c` variant selection.** MPI 4.0 introduced `_c` variants
   of most communication functions (e.g., `MPI_Send_c`, `MPI_Bcast_c`) that
   accept `MPI_Count` instead of `int` for element counts, enabling messages
   larger than `INT_MAX` elements. Every shim in `csrc/ferrompi.c` that accepts
   a count selects the `_c` variant at compile time via `#if MPI_VERSION >= 4`
   and falls back to the classic variant on MPI 3.x. Rust callers see a single
   stable signature in all cases; the version branch is invisible.

3. **`MPI_UNDEFINED` normalization.** MPI uses `MPI_UNDEFINED` (an
   implementation-defined negative sentinel) to indicate that a rank is not
   a member of a group or that a communicator color was unspecified. The C
   shims normalize `MPI_UNDEFINED` to `-1` before returning, giving Rust a
   stable sentinel that does not vary across MPI implementations.

4. **`install_errors_return` on comm-creating shims.** Every shim that
   produces a new communicator (`ferrompi_comm_dup`, `ferrompi_comm_split`,
   `ferrompi_comm_create`, `ferrompi_create_from_group`, etc.) calls the
   internal `install_errors_return(newcomm)` helper before returning. This
   sets `MPI_ERRORS_RETURN` as the error handler on the new communicator,
   ensuring that MPI errors surface as return codes rather than aborting the
   process. Centralizing this in C means Rust callers cannot forget it.

5. **Op trampolines.** `MPI_Op_create` requires a plain C function pointer;
   it provides no `void* user_data` argument. To route a call back into the
   correct Rust closure, the C layer provides `MAX_OPS` (16) distinct trampoline
   functions generated by a preprocessor macro, each with its slot index baked
   into its function body. This pattern is described fully in ADR-0005
   (`docs/adr/0005-mpi-op-create.md`).

6. **Runtime-queried `MPI_MODE_*` constants.** RMA window assertion flags
   (`MPI_MODE_NOSTORE`, `MPI_MODE_NOPUT`, `MPI_MODE_NOPRECEDE`,
   `MPI_MODE_NOSUCCEED`, `MPI_MODE_NOCHECK`) are integer constants whose
   values are defined by the MPI implementation header, not by the standard.
   The C layer exposes `ferrompi_win_fence_mode_values` and
   `ferrompi_win_pscw_mode_values` shims that return these constants at
   runtime. Rust caches them once per process in `OnceLock`-protected arrays.
   This prevents hardcoding implementation-specific integers in Rust source.

### Why bindgen alone is insufficient

`bindgen` reads C headers and produces Rust `extern "C"` declarations with
types derived from the C type system. For MPI, this means producing Rust
types corresponding to `MPI_Comm`, `MPI_Request`, `MPI_Win`, and so on.

The problem is that these types are not ABI-stable across MPI implementations:

- **MPICH** (including its derivatives: Cray MPICH, Intel MPI, MVAPICH2)
  defines `MPI_Comm` as `int`, `MPI_Request` as `int`, and all other handle
  types as integer typedefs. The MPI object is identified by an index into a
  process-global table inside the MPI library.

- **Open MPI** defines `MPI_Comm` as `struct ompi_communicator_t*`,
  `MPI_Request` as `struct ompi_request_t*`, and similarly for other handles.
  The MPI handle is a pointer to an allocated C struct.

A single Rust binary compiled against MPICH-generated bindgen output will have
`MPI_Comm = i32` throughout; the same source compiled against Open MPI headers
will have `MPI_Comm = *mut ompi_communicator_t`. These are different types with
different sizes, alignments, and calling conventions. Distributing a
`*-sys` crate that encodes this type in its public API means the crate's ABI
changes depending on which MPI implementation the user's system provides. A
downstream crate that stores `MPI_Comm` values cannot be compiled once and
linked against either implementation.

Beyond the handle type problem, the `_c` large-count variants are absent from
MPI 3.x headers entirely: bindgen would produce bindings for either the 3.x
API or the 4.x API depending on what header it finds, with no mechanism to
produce a single stable Rust API that works across both. And the op-trampoline
problem — needing `MAX_OPS` distinct function pointers with no `user_data`
argument — requires either C-level code generation or unsafe Rust macro
gymnastics that are harder to audit.

## Decision Drivers

1. **ABI portability across MPI implementations.** A single ferrompi build
   must link correctly against MPICH, Open MPI, and Cray MPICH without
   recompilation or conditional compilation in Rust source. The C wrapper
   normalizes all MPI handle types to `int32_t` or `int64_t` slot indices
   before they cross the FFI boundary, producing a stable type at the Rust
   level regardless of the underlying MPI handle representation.

2. **Encapsulation of version-gated features.** MPI 4.0 introduced large-count
   `_c` variants that cannot be referenced in Rust source if the build
   targets MPI 3.x — any attempt to name `MPI_Send_c` in Rust FFI declarations
   on a pre-4.0 system would fail to link. By placing the `#if MPI_VERSION >= 4`
   conditional in C, the branch is resolved at the C compilation stage. Rust
   callers see a single stable entry point in all cases.

3. **Centralized handle management.** Exposing raw `MPI_Comm` or `MPI_Request`
   values to Rust would require Rust code to be written differently depending
   on whether handles are integers (MPICH) or pointers (Open MPI). The handle
   tables in the C layer absorb this variation: Rust always receives an
   `int32_t` slot index, manipulates it freely, and passes it back to the C
   layer to retrieve the underlying MPI handle.

4. **Pre-flight FFI callback infrastructure.** `MPI_Op_create` provides no
   mechanism for attaching user data to a function pointer. The only correct
   solution is multiple distinct function pointers, one per slot, each
   hard-coded to retrieve its associated closure by index. This is naturally
   expressed in C as a preprocessor macro that stamps out `MAX_OPS` functions.
   Producing equivalent code in Rust would require procedural macros and
   `unsafe` pointer arithmetic that is more complex to audit than the
   straightforward C macro expansion. See ADR-0005.

5. **Minimum unsafe surface in Rust.** Every call to a `bindgen`-generated
   MPI binding would be an `unsafe` call in Rust, because the raw MPI API
   manages memory and handle lifetimes in ways that Rust cannot verify
   statically. Concentrating the unsafe behavior in a single C translation unit
   that is reviewed once — with a stable, typed `int32_t`-based interface
   exposed to Rust — means the unsafe surface in Rust is a narrow FFI module
   (`src/ffi.rs`) rather than scattered `unsafe` blocks at every call site in
   safe wrapper code.

## Options Considered

### Option A (chosen): Hand-written C wrapper with handle-table pattern

A hand-written C layer (`csrc/ferrompi.c` and `csrc/ferrompi.h`) that
translates between the MPI implementation's native handle types and
fixed-width integer slot indices. The C layer owns the handle tables,
encapsulates version-conditional code, and provides the op trampolines.
Rust binds to this C layer via `src/ffi.rs`; safe Rust wrappers sit above
the FFI declarations.

**Pros:**

- _ABI stability._ The Rust-visible API uses only `int32_t`, `int64_t`,
  `bool`, and `void*` types. These are the same size and alignment on every
  target and every MPI implementation. No conditional compilation is required
  in Rust source.
- _Version transparency._ The `#if MPI_VERSION >= 4` branches in C mean that
  the same Rust source file compiles and works correctly whether the build
  host provides MPI 3.x or MPI 4.x. The Rust API does not need to model MPI
  version differences.
- _Low Rust unsafe surface._ The `unsafe` in Rust is confined to `src/ffi.rs`
  (the `extern "C"` block with raw function signatures) and the immediate
  safe wrappers that convert FFI results to `Result<T, Error>`. All Rust code
  above this layer is safe.
- _Custom infrastructure._ The handle tables, the `install_errors_return`
  automation, the `MPI_UNDEFINED → -1` normalization, and the op trampolines
  are all straightforward in C and would be significantly more complex or
  less auditable if expressed purely in Rust.
- _Single review unit._ A security audit of ferrompi's FFI layer reads one C
  file and one C header. The invariants — correct slot bounds, null-sentinel
  checks, error-handler installation — are enforced in one place.

**Cons:**

- _Code volume._ The C layer is 4629 LOC in `csrc/ferrompi.c` and 2171 LOC
  in `csrc/ferrompi.h`. Every new MPI function requires additions to both
  files, to `src/ffi.rs`, and to the safe Rust wrapper, following the
  six-layer convention documented in `epic-07-summary.md`.
- _Manual maintenance._ When the MPI standard adds new functions or when
  ferrompi adds support for new MPI versions, the C shims must be written by
  hand. There is no automated mechanism to detect that a new MPI function
  exists and has not yet been wrapped.
- _Dual-language code review._ Any change that touches the FFI boundary
  requires reviewers to read both Rust and C. Contributors who are fluent in
  Rust but unfamiliar with C face a higher onboarding cost for FFI changes.

### Option B (rejected): Pure bindgen + safe Rust wrappers

Use `bindgen` to generate a `ferrompi-sys` crate from the system MPI headers.
Layer safe Rust wrappers on top, using Rust `cfg` attributes (`#[cfg(mpi_version = "4")]`)
to gate large-count variants.

**Pros:**

- _Less hand-written code._ The `*-sys` crate is generated automatically; the
  surface area of manually maintained code shrinks to the safe wrapper layer.
- _Automatic updates._ When MPI implementations add new functions, running
  `bindgen` regenerates the `*-sys` crate to expose them without any manual
  intervention.
- _Community alignment._ The `bindgen` + `*-sys` pattern is the established
  Rust convention for C library bindings and is familiar to most Rust
  contributors.

**Cons:**

- _ABI instability._ This is the fundamental rejection criterion. `bindgen`
  encodes the MPI implementation's native handle types into the generated Rust
  types. A crate compiled against MPICH headers has `type MPI_Comm = i32`; the
  same crate compiled against Open MPI headers has `type MPI_Comm = *mut
ompi_communicator_t`. The Rust ABI of the generated crate changes with the
  MPI implementation. Downstream crates that store `MPI_Comm` values in data
  structures, pass them across function boundaries, or serialize them cannot
  be written portably against the `*-sys` crate. ferrompi's explicit goal is
  to provide a single Rust API that links against MPICH, Open MPI, and Cray
  MPICH without source changes.

- _No solution to version gating._ `bindgen` generates bindings for the headers
  it is given. If the build host provides MPI 3.1 headers, the generated crate
  contains no `MPI_Send_c` declaration. If it provides MPI 4.0 headers, the
  crate contains `MPI_Send_c` but not the 3.x fallback. A `cfg`-attribute
  approach requires the build system to detect the MPI version, define a
  feature flag, and condition every call site in the safe wrapper on that flag.
  This scatters version-awareness throughout the Rust codebase. The C layer
  solves this with a single `#if MPI_VERSION >= 4` at the point of the call,
  invisible to callers.

- _FFI callbacks still require unsafe C._ The op-trampoline problem (see
  Decision Driver 4 and ADR-0005) cannot be solved without multiple distinct
  C-level function pointers. A pure-bindgen approach would still require a
  hand-written C file for the trampolines, negating much of the code-reduction
  argument.

### Option C (rejected): Wrap an existing C or C++ MPI abstraction library

Use an existing higher-level MPI C/C++ abstraction — for example, Boost.MPI,
MPL (a header-only C++17 MPI wrapper), or OOMPH (Object-Oriented MPI for
HPC) — as an intermediary. Rust would bind to the abstraction library rather
than to MPI directly.

**Pros:**

- _Leverage existing abstractions._ These libraries already handle many of the
  same problems: type-safe message passing, communicator wrappers, reduction
  operator registration.
- _Community validation._ Libraries like Boost.MPI are battle-tested across
  many HPC environments and MPI implementations.

**Cons:**

- _Heavy dependency._ Boost.MPI requires Boost (a large C++ library) and a
  C++ compiler in the build graph. MPL is header-only but still requires C++17.
  ferrompi's current build graph is Rust + C with no C++ toolchain requirement.
  Adding Boost or a C++ dependency for every downstream user of ferrompi is a
  significant ask.

- _Opinionated API surface._ Abstraction libraries encode design choices about
  communicator ownership, buffer management, and collective patterns that may
  conflict with ferrompi's API goals. Adapting ferrompi's `Communicator` type
  to sit above Boost.MPI's `communicator` would require either forgoing
  ferrompi's API design or writing another translation layer on top of the
  abstraction — two C/C++ layers instead of one.

- _Does not solve the core problem._ The handle-type ABI problem does not
  disappear when Boost.MPI is in the stack: Boost.MPI itself is compiled
  against a specific MPI implementation, and any raw MPI handle that crosses
  the Boost.MPI-to-Rust boundary still carries the implementation's type.
  The only way to eliminate the problem is to convert to fixed-width integer
  indices at the MPI boundary, which is exactly what the ferrompi C layer does.

- _Version gating moves, not disappears._ The `_c` large-count variants and
  MPI 4.0 features would need to be gated at the C++ abstraction level. Unless
  the chosen library already handles this, the version-gating work remains.

## Decision

**Option A (hand-written C wrapper) is chosen.**

Rationale against each decision driver:

- **Driver 1 — ABI portability.** The handle-table pattern converts every MPI
  handle type to an `int32_t` slot index before it crosses the FFI boundary.
  The Rust API is `int32_t`-based throughout and is identical regardless of
  whether the underlying MPI implementation represents `MPI_Comm` as an integer
  or a pointer. No `cfg` attribute, no conditional type alias, no per-implementation
  build flag is required anywhere in Rust source.

- **Driver 2 — Version-gated features.** Every communication shim in
  `csrc/ferrompi.c` has the form:

  ```c
  #if MPI_VERSION >= 4
      return MPI_Send_c(buf, (MPI_Count)count, dt, dest, tag, comm);
  #else
      return MPI_Send(buf, (int)count, dt, dest, tag, comm);
  #endif
  ```

  This branch is resolved once, at C compilation time. The Rust function
  signature for `ferrompi_send` does not change; neither does its body. Adding
  support for a new MPI version means editing one block of C, not auditing
  every call site in Rust.

- **Driver 3 — Centralized handle management.** All seven handle tables are
  defined in one file (`csrc/ferrompi.c`), with a single `init_tables`
  function, consistent `alloc_*`/`free_*`/`get_*` naming, and uniform
  occupancy tracking. Adding a new handle type follows the same pattern.
  Rust code that calls `alloc_comm` receives a `int32_t`; it does not need
  to know whether that value will ultimately be used as an MPICH integer or
  an Open MPI pointer.

- **Driver 4 — FFI callback infrastructure.** The 16 op trampolines are
  generated by a single C preprocessor macro and audited once. Adding a
  seventeenth (if `MAX_OPS` is increased) requires one integer change. The
  equivalent in Rust would require either a procedural macro (adding a build
  dependency and a non-trivial amount of unsafe pointer casting logic) or
  a repeated block of `unsafe extern "C" fn` definitions that is harder to
  keep consistent than a `#define` loop in C.

- **Driver 5 — Minimum unsafe surface.** The `unsafe` in Rust is confined to
  `src/ffi.rs`. Safe wrapper functions in `src/comm/`, `src/window.rs`,
  `src/op.rs`, and `src/lib.rs` do not contain `unsafe` blocks beyond the
  necessary FFI call sites; they operate on `int32_t` handles and `Result`
  types. A contributor auditing ferrompi for memory safety reads `csrc/ferrompi.c`,
  `csrc/ferrompi.h`, and `src/ffi.rs` — a bounded, well-defined surface.

Option B is rejected because `bindgen` output is implementation-specific:
the generated `MPI_Comm` type is not the same between MPICH and Open MPI, and
no amount of safe Rust wrapping can reconcile two binaries with different
type-level definitions of the same handle type. The version-gating and
op-trampoline problems also remain unsolved.

Option C is rejected because every candidate abstraction library either adds
a heavy C++ dependency, encodes opinionated API choices incompatible with
ferrompi's design, or defers rather than eliminates the ABI portability problem.

## Consequences

**The C wrapper grows with every new MPI feature.** Each addition follows the
six-layer convention: C declaration in `csrc/ferrompi.h`, C implementation in
`csrc/ferrompi.c`, `extern "C"` declaration in `src/ffi.rs`, safe Rust wrapper,
integration example, and a test entry in `tests/run_mpi_tests.sh`. This is a
deliberate constraint, not an oversight: the six-layer structure ensures that
every new MPI surface has a corresponding test and a documented interface.

**Reviewers must read both Rust and C for FFI changes.** A change that modifies
a C shim signature must be reflected in `src/ffi.rs`; a change to handle-table
capacity constants in C must be coordinated with any Rust code that assumes
bounds on slot indices. This dual-language review requirement is the primary
ongoing cost of the C wrapper approach.

**The C layer is the security-audit surface.** `csrc/ferrompi.c` and
`csrc/ferrompi.h` are the only files where unchecked pointer arithmetic,
raw `MPI_*` handle manipulation, and unsafe C operations occur. The size of
this surface must be kept bounded; any refactoring that moves logic from Rust
into C for convenience (rather than necessity) should be resisted.

**Version gating happens in C; Rust does not need version-specific code paths.**
New MPI versions that add `_c` large-count variants or new API functions are
accommodated by editing `#if MPI_VERSION` blocks in C. The consequence is that
`csrc/ferrompi.c` will accumulate more version branches over time, but Rust
source remains stable. Rust-side version detection (`Mpi::version()` and
`OnceLock`-cached runtime probes) is reserved for behavioral differences that
cannot be resolved at compile time — not for API-surface differences.

**Handle tables are fixed-size; capacity constants are tuning parameters.**
`MAX_COMMS`, `MAX_REQUESTS`, `MAX_WINDOWS`, `MAX_INFOS`, `MAX_GROUPS`,
`MAX_DATATYPES`, and `MAX_OPS` are defined at the top of `csrc/ferrompi.c`.
Applications with unusual workloads (e.g., a program that creates thousands of
communicators) may need to increase these values and recompile. This is an
accepted trade-off against the complexity of a dynamic allocation strategy
inside the C layer. The request table (`MAX_REQUESTS = 16384`) is sized for
high in-flight non-blocking operation counts; the op table (`MAX_OPS = 16`) is
sized for the realistic number of distinct user-defined reduction operations.

**The slot-claim concurrency strategy for the request table is specified in
ADR-0002** (`docs/adr/0002-handle-tables.md`). The C11 atomic CAS pattern
used there is the intended model for hardening the remaining plain-`int` tables
(comm, win, info) in future work. ADR-0002 binds that future implementation.

**The trampoline scheme for user-defined ops is specified in ADR-0005**
(`docs/adr/0005-mpi-op-create.md`). The `ferrompi_user_op_trampoline_N`
functions in `csrc/ferrompi.c`, the `op_closure_data`/`op_closure_vtbl` arrays,
and the drop ordering in `Drop for UserOp<T>` are all consequences of the
decision made here to centralize FFI callback infrastructure in C.

## Alternatives Considered

Two further alternatives were evaluated informally and not given full
option treatment:

**Directly generated FFI via the `cc` crate with a minimal bridge.** The `cc`
crate can compile C source during `cargo build`. An alternative to a
hand-written wrapper would be a `build.rs` script that queries MPI flags
(`mpicc --showme:compile`, `mpicc --showme:link`), generates a thin `.c` file
from a template (parameterized by MPI version and implementation), and
compiles it. This approach would automate some of the boilerplate but would
require a code-generation step that is harder to read and audit than a static
C file. The handle-table pattern, op trampolines, and version branches would
still need to be expressed in the generated C — the reduction in human-written
lines would be modest at best.

**Single-implementation builds.** ferrompi could accept that a given binary
links against exactly one MPI implementation, distribute separate `ferrompi-mpich`
and `ferrompi-openmpi` crates, and use Cargo feature flags to select the right
one. This eliminates the ABI portability requirement and allows `bindgen` output
to be used directly. It was rejected because HPC clusters frequently present
multiple MPI implementations (system MPI, user-installed MPICH, Cray MPICH),
and requiring users to recompile ferrompi when switching implementations is an
unacceptable operational friction. The handle-table approach pays a modest code
cost to eliminate this friction entirely.
